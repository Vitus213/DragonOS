# 主分支对齐 Code Review

## 目标

在合并前识别阻塞问题并给出可执行修复建议，确保变更满足质量门禁。

## 工作步骤

1. 识别基线分支（`main` 或 `master`）并查看 `base...HEAD` 差异。
2. 审查提交粒度与提交信息质量（是否解释为什么）。
3. 重点审查：
   - Asterinas 参考语义一致性
   - 错误码与异常路径
   - 并发安全（锁顺序、竞态窗口）
   - 资源生命周期（引用计数、释放路径）
   - 性能与回归风险
4. 结合测试结果给出合并结论。

## 输出格式

```markdown
## Review 结论

### 1) 阻塞问题（Must Fix）
| 级别 | 问题 | 影响 | 修复建议 | 证据 |
|---|---|---|---|---|

### 2) 非阻塞建议（Should Fix）
| 问题 | 建议 | 证据 |
|---|---|---|

### 3) 已确认优点
- ...

### 4) 合并前清单
- 语义一致性通过
- 测试通过（新增 + 回归）
- 文档同步完成
- 无明显性能/安全倒退
```

## 质量要求

- 结论必须可验证，且附 `file:line` 或命令证据。
- 区分阻塞与非阻塞，避免泛泛而谈。
- 给出具体修复方向，不只指出问题。
- 若仓库存在 `flake.nix`，命令证据应优先来自 `nix develop -c <command>` 环境。
