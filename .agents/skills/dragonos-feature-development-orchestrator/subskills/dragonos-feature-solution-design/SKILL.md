# 方案设计与代码落地

## 目标

把调研结论转化为可执行、可验证、可合并的实现/修补方案，并完成代码落地。

## 输入依赖

- DragonOS 现状调研结果
- Asterinas 参考语义基线结果

若输入缺失，先列出假设并标注影响范围。

## 工作步骤

1. 明确目标与非目标，定义 MVP 与后续增强阶段。
2. 设计抽象边界：模块职责、数据结构、状态机、错误模型。
3. 给出文件级改动计划（先基础设施，后功能接线，最后测试）。
4. 先给出关键代码草案，再在对应文件中实施改动。
5. 运行必要验证（至少语法/构建级检查），记录结果。
6. 给出测试矩阵与回滚策略。

### 构建环境优先级

- 若项目根目录存在 `flake.nix`，优先在 Nix 开发环境执行验证命令：`nix develop -c <command>`。
- 本项目默认构建命令应写为：`nix develop -c make kernel`。
- 若 `nix develop` 失败，可回退到宿主环境命令，但必须在输出中说明回退原因。

## 质量要求

- 明确跨模块依赖顺序，避免循环依赖。
- 禁止以“兼容分支/特判绕过”代替根因修复。
- 覆盖异常路径、边界条件与回归风险。
